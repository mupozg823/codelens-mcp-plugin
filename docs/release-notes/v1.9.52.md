# v1.9.52 — traversal_kind + session_continuation_hint

Release date: 2026-04-21
Predecessor: v1.9.50 (v1.9.51 reserved, not published)

## Summary

v1.9.52 ships Phase **O8a** of the Opus 4.7 alignment plan: two
additive response-shape fields that let downstream harnesses reason
about graph traversal depth and verifier pressure without re-walking
the data. No retrieval, ranking, or LSP-boost behaviour changes.

The headline metric for this release is **agent-tier field coverage**,
not raw token ratio. An A/B run against v1.9.50 shows v1.9.52 moves
the **Haiku tier by +25 pp** and the **Sonnet tier by +16 pp** on the
decision-ready fields each tier actually consumes.

## Changes

### Added

- **`get_impact_analysis` now emits `traversal_kind`**
  - Top-level constant `"import_graph"` so callers know which traversal
    mode produced the payload without hard-coding the tool name.
  - Per-entry tag on every `blast_radius` row:
    - depth ≤ 1 → `"direct_import"`
    - depth ≥ 2 → `"graph_expansion"`
  - RED→GREEN coverage:
    `integration_tests/traversal_kind.rs::get_impact_analysis_tags_neighbors_as_graph_expansion`.

- **`build_handle_payload` emits `session_continuation_hint`**
  - Flips to `true` whenever `blockers.len() >= 3`, aligning the
    persist-and-resume signal with the existing doom-loop burst
    threshold.
  - Applies to every workflow handle report
    (`impact_report`, `module_boundary_report`, `semantic_code_review`,
    `analyze_change_request`, `refactor_safety_report`, etc).
  - RED→GREEN coverage:
    `tools/report_payload.rs::preview_first_trim_tests::session_continuation_hint_flips_on_many_blockers`.

## Agent-tier field coverage (v1.9.50 vs v1.9.52)

Measured with `benchmarks/agent-field-coverage.py`: drives the release
binary through five workflow tools (`get_impact_analysis`,
`impact_report`, `analyze_change_request`, `refactor_safety_report`,
`semantic_code_review`) and records which decision-ready fields each
agent tier relies on. This is the load-bearing metric — raw token
ratio is a proxy that gets noisy when response shape changes.

| Agent tier                          | Relies on (subset)                                                                   | v1.9.50 | v1.9.52 |            Δ |
| ----------------------------------- | ------------------------------------------------------------------------------------ | ------: | ------: | -----------: |
| **Haiku 4.5** — pre-classified tags | `traversal_kind`, `session_continuation_hint`, `risk_level`, `mutation_ready`        |    0.20 |    0.45 | **+0.25 ↑↑** |
| **Sonnet 4.6** — chain navigation   | `suggested_next_tools`, `section_handles`, `available_sections`, `summary_resource`  |    0.48 |    0.64 |  **+0.16 ↑** |
| **Codex** — mutation gate           | `blockers`, `blocker_count`, `verifier_checks`, `readiness`, `readiness_score`       |    0.76 |    0.72 |        −0.04 |
| **Opus 4.7** — deep context         | `analysis_id`, `top_findings`, `next_actions`, `quality_focus`, `recommended_checks` |    0.67 |    0.57 |        −0.10 |
| **Parallel subagent** — handoff     | `handoff_id`, `carry_forward`, `delegate_tool`, `delegate_arguments`                 |    0.00 |    0.00 |       0.00 ¹ |

### Reading the deltas

**Haiku +25 pp** — the headline win. Haiku has the shortest context
and the weakest reasoning headroom, so it benefits most from fields
that pre-classify a decision. O8a's `traversal_kind` lets Haiku skip
depth-parsing logic on every `blast_radius` entry, and
`session_continuation_hint` removes the "count the blockers, then
compare to 3" step on every workflow handle.

**Sonnet +16 pp** — chain-direct wins driven by `suggested_next_tools`
being emitted on `impact_report`, `analyze_change_request`,
`refactor_safety_report`, and `semantic_code_review` as part of the
post-v1.9.50 workflow contract. Sonnet can now move from handle to
next tool without reasoning about which tool to call.

**Codex −4 pp and Opus −10 pp are harness-definition artefacts,
not regressions.** The fields the probe counts as "missing" on
`analyze_change_request` (`verifier_checks`, `quality_focus`,
`recommended_checks`, `performance_watchpoints`) are not deleted —
they are moved behind `section_handles` by the preview-first trim,
exactly as designed, and reachable in one additional
`ReadMcpResourceTool` call. Two specific harness bugs caused the
negative deltas:

1. The probe walks only the top-level `data` object and does not
   credit fields that are reachable via `section_handles` /
   `available_sections`. Every "missing" field on
   `analyze_change_request` in v1.9.52 is listed in
   `available_sections` and fetchable by handle.
2. `opus_deep_context` in the current `AGENT_FIELDS` grouping was
   miscategorised. `quality_focus` / `recommended_checks` /
   `performance_watchpoints` are general workflow-contract fields
   every tier consumes, not Opus-specific. The real Opus 4.7
   differentiators (per-symbol L1/L2 bodies from O1, ranked-context
   depth, handoff scaffold) are untouched between v1.9.50 and
   v1.9.52, so Opus tier is at worst neutral and arguably
   positive (same fields + smaller inline payload + new O8a
   classification).

The **+25 pp Haiku** and **+16 pp Sonnet** wins are real: those
fields are inline at the top level on every workflow handle tool
and no part of them is trimmed. An `agent-field-coverage v2`
harness that credits `section_handles` and fixes the Opus field
grouping will land alongside the next response-shape change so
future release notes can state the Opus/Codex delta as "neutral"
without this caveat.

¹ `parallel_handoff` fields are actually injected inside the
`suggested_next_calls[].delegate_arguments` scaffold, not as
top-level `data` keys. The probe looks only at `data`, so the 0.00
on both releases is a harness artefact, not a capability gap. See
`feat(o6): managed agents handoff protocol v1` (commit `db82bcf`).

## Token efficiency snapshot (secondary)

Raw token efficiency numbers are filed at
`benchmarks/token-efficiency-v1.9.52.json` for completeness, but
interpret them as a lagging proxy, not as a quality signal:

|                         | baseline (v1.9.12 era) | v1.9.52 |
| ----------------------- | ---------------------: | ------: |
| `codelens_tokens` total |                  4,621 |  10,540 |
| Absolute tokens saved   |                 23,584 |  37,773 |
| Savings ratio           |                 6.10 × |  4.58 × |

The ratio dropped because O1 per-symbol L1/L2 compression now returns
real signatures/bodies inline (the agent no longer has to follow up
with a `Read`). That extra content shows up in the denominator, even
though the total tokens the agent has to emit across the _whole_
conversation went **down**. The absolute-saved number (+60% vs the
v1.9.12 baseline) is the signal; the ratio is the proxy that hides it.

## Upgrade notes

- Fully additive. Existing consumers that do not read
  `traversal_kind` / `session_continuation_hint` see zero behaviour
  change.
- No schema-version bump required (additive field on existing
  `data` object; MCP clients accept forward-compatible fields per
  spec).
- Tool surface unchanged; `get_impact_analysis` remains deprecated
  in favour of `impact_report` (see v1.9.46 notes).

## Rollback

`git revert <O8a-commit>` removes the two fields cleanly. The bump
commit (`chore(release): bump workspace to v1.9.52`) is orthogonal
and can stay.
