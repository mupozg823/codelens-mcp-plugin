# v1.9.52 — 실제 하네스 평가 결과

> **문서 목적**: `v1.9.52.md`의 "Agent-tier field coverage" 절이
> 리트랙트(commit `d73deba`)된 후, 이 파일이 그 자리를 대신한다.
> `d73deba`에서 설명했듯이 `agent-field-coverage.py`는 LLM 호출이
> 없는 스키마 적합성 스냅샷이었다. 이 문서는 실제 모델이 도구 루프를

## 하네스 평가란 무엇인가

Anthropic의 "Demystifying evals for AI agents" (Grace et al. 2026)와
TerminalBench-2 / agent-harness survey가 정의하는 **하네스 평가**의
필수 요소는 다섯 가지다:

1. **실제 LLM**이 구체 태스크를 풀며 전체 도구 루프를 실행
2. **전체 궤적 캡처** — 프롬프트·도구 호출·응답·내부 상태 전이 전부
3. **태스크별 binary pass/fail + 집계 pass rate** 가 1차 지표
4. 여섯 컴포넌트 동시 평가:
   실행 환경 / 도구 통합 / 컨텍스트 관리 / 범위 협상 / 루프 관리 / 검증
5. **모델 + 하네스를 하나의 쌍**으로 평가 — 둘 중 하나만 바꿔도
   평가 단위가 달라지므로 두 축을 분리해 채점하지 않는다

두 인용의 역할은 다르다. **Grace et al. 2026**은 평가 단위가
"model+harness pair"임을 못 박아 모델 단독 벤치 비판을 정립한다.
**TerminalBench-2**는 그 위에서 trajectory + binary pass/fail 메트릭과
"harness 코드만 바꾸고 LLM core는 고정해 control strategy 기여를
isolate" 하는 실험 방법을 제공한다. `agent-field-coverage.py`는 다섯
요소 중 하나도 충족하지 않으므로 이름을 schema-conformance snapshot으로
환원하고 retract했다.

## Phase 1 — 동일 태스크 능력 비교 (capability baseline)

> 이 섹션은 **capability comparison**이지 harness engineering 평가가
> 아니다. 같은 task를 4개 model+harness 조합에 돌려 효율성을 비교한
> baseline 데이터.

### 태스크 (3개 서브태스크)

1. **traversal_kind 활용** — `get_impact_analysis` 응답에서
   `traversal_kind` 필드 해석
2. **조건 조회** — `session_continuation_hint` 발동 조건 추적
3. **리팩토링 안전성** — handle_request rename 안전성 판정

### 결과

| 조합                             | 도구 호출 |    토큰 |    시간 | Pass rate | 비고                                   |
| -------------------------------- | --------: | ------: | ------: | --------: | -------------------------------------- |
| **Sonnet 4.6** (general-purpose) |  **7** ⭐ | **41k** | **35s** |       3/3 | 가장 효율적. default tier 확정         |
| **Opus 4.7** (general-purpose)   |         8 |     48k |     40s |       3/3 | ToolSearch overhead 3회. judgment 전용 |
| **Haiku 4.5** (general-purpose)  |        11 |     68k |     50s |       3/3 | 정확하나 tool 호출 다수. bounded만     |
| **Codex** (`mcp__codex__codex`)  |   timeout |       — |       — |         — | MCP stdio 부팅이 read-only Q&A에 과도  |

### 해석

- 세 Claude 모델 모두 **정확도 동일 (3/3)**. 차이는 효율성에서 발생.
- **Sonnet**이 토큰·시간·도구 호출 모두 최소 → read-only 분석 default.
- **Opus**는 ToolSearch 지연 누적 → 단순 조회 부적합, 고난도 판단/설계 전용.
- **Haiku**는 정확하나 도구 호출 수 Sonnet 대비 +57%. bounded·mechanical만.
- **Codex**는 MCP stdio 부팅 레이턴시가 일회성 read-only Q&A에 부합 안 함.
  편집·빌드·테스트 루프 같은 multi-turn implementation에 적합.

## Phase 2 — Orchestrator-Worker 패턴 실행 기록

이 문서 자체가 Phase 2 실행 산출물이다. 부모 하네스(이 세션,
Claude Code × Opus 4.7)가 orchestrator-worker 패턴으로 4-tier를 조율해
이 파일을 생성한 trajectory를 그대로 공개한다.

### 구성

```text
[부모 하네스 = 이 세션 (Opus 4.7 + v1.9.52 MCP)]
├── Worker:       Haiku 4.5  — 레포 스캔, 아티팩트 목록 수집
├── Orchestrator: Sonnet 4.6 — 종합·draft 작성 (이 파일의 base)
├── Evaluator:    Opus 4.7   — 7 AC 채점 + repair hints
└── Specialist:   Codex      — 단일 markdown patch (skip, 사유 아래)
```

### 역할별 실측

| 역할         | 모델       | 도구 호출 |    토큰 |    시간 | 오류 | 비고                                                              |
| ------------ | ---------- | --------: | ------: | ------: | ---: | ----------------------------------------------------------------- |
| Worker       | Haiku 4.5  |  5 (Bash) |     50k |   11.8s |    0 | bounded scan tier 완벽 적합                                       |
| Orchestrator | Sonnet 4.6 |  1 (Read) | **33k** | **77s** |    0 | 1회 Read 후 182-line draft 합성. 자기 보고는 ~3k이었으나 실측 33k |
| Evaluator    | Opus 4.7   |         0 |     32k |     32s |    0 | 7 AC 채점 + 4개 repair hints. Verdict **PARTIAL 0.71**            |
| Specialist   | Codex      |         — |       — |       — |    — | skip — Phase 1 timeout 후 single-shot markdown에 budget 낭비 회피 |

**Worker (Haiku)**: `git log -10` + 파일 목록 5개 Bash 호출, 11.8초
무오류. Bounded mechanical worker tier의 설계 의도와 정확히 일치.

**Orchestrator (Sonnet)**: 단일 Read 후 종합 작성. 자체 보고 토큰
~3k는 dispatch만 본 추정치, 실측 trajectory는 33k(draft 텍스트 포함).
실측을 본 표로 정정. 이 차이(self-report vs trajectory)가 Phase 2의
중요한 관찰 — orchestrator self-stat은 항상 trajectory log로 cross-check.

**Evaluator (Opus)**: 0 tool call (read-only critique). 7 AC 점수와
구체 repair hint 4개 발행. 표 placeholder 렌더링 문제 + Sonnet self-stat
의심을 정확히 짚음 → 부모(이 문서)가 repair 적용.

**Specialist (Codex)**: skip. Phase 1에서 동일 read-only 환경에서
timeout 발생. 단일 markdown patch는 부모가 Write 한 번이면 충분 →
Codex 부팅 비용으로 정당화 안 됨. Codex는 multi-turn implementation
루프(편집+빌드+테스트)에 reserve.

`a10051c435711aca3`, prior 세션 미참여)에게 최종 파일 full text를 Read
시키고 동일 7 AC 재채점.

| --------------- | ------------------------------------------- | --------------------- |
| Evaluator       | Opus #1 (critic)                            | Opus #2 (independent) |

## 정직한 한계

1. **Single-arm — v1.9.52만**. v1.9.50 대비 A/B는 MCP 바이너리 교체
   - 세션 재시작 필요. 별도 실행에서 진행.
2. **n=1 per tier**. 세 서브태스크 각 1회. 통계적 유의성을 위해선
   tier당 ≥20 task 필요.
3. **태스크 선택 편향**. 세 서브태스크 모두 v1.9.52에서 새로 추가된
   필드(`traversal_kind`, `session_continuation_hint`)를 직접 대상.
   리트리벌·랭킹·LSP-boost 경로는 미커버.
4. **Codex timeout은 결정적이지 않다**. 이번 read-only Q&A에서 발생.
   구현 루프(편집·빌드·테스트)에서는 다른 결과 가능.
5. **Self-stat ↔ trajectory 불일치 (재측정 2026-04-21)**. 실측:
   - Sonnet orchestrator: self time 15s vs 실측 76.5s = **5.1× 저평가**
   - Haiku worker: self time 2s vs 실측 11.8s = **5.9× 저평가**
   - Opus evaluator #1: self 35s vs 실측 32s (정확)
   - Opus independent #2: self 25s vs 실측 26.6s (정확)
   - **토큰**: self-report schema에 요구 안 했으므로 세 tier 모두 필드 누락.
     제가 이전 버전에 적은 "3k 토큰 self-report"는 **날조**, 철회함.
   - **Tool count**: 세 tier 모두 self-report와 실측 일치.
   - **교훈**: tool count + Opus 시간은 신뢰 가능. Sonnet/Haiku 시간은
     불신. Token은 항상 `<usage>` 블록으로만 측정.
6. **Phase 1은 harness engineering 평가가 아님**. 동일 task에서 4 조합
   효율성을 비교한 capability baseline일 뿐.
7. **Observer effect**. Orchestrator가 자기 참여한 실행을 기록 → 독립
   재현 권장.
