# CodeLens MCP v1.9.14 — Release Notes

> **Release date**: 2026-04-13
> **Scope**: Retrieval adaptation cleanup, external benchmark smoke coverage, and product-facing documentation resync.

---

## Why this release

The immediate gap was no longer "add more tools." It was drift:

1. the promoted retrieval story mixed generic improvements with CodeLens-specific vocabulary shaping
2. public benchmark claims were still dominated by self-repo numbers
3. deployment and release documentation lagged behind the actual published channels and current binary targets

v1.9.14 tightens that surface. It keeps the existing tree-sitter-first and bounded-workflow architecture, but makes the product story more honest and more operable.

---

## Highlights

### 1. Generic vs project-specific retrieval adaptation is explicit

The retrieval stack is now described and implemented as a two-tier system:

- **generic adaptation** stays in code and is intended to transfer across repositories
- **project-specific adaptation** is isolated in `.codelens/bridges.json`

That separation matters because CodeLens-specific vocabulary can help local benchmarks while saying nothing about generalization. v1.9.14 keeps the local optimization path, but stops treating it as universal evidence.

### 2. Project bridges can be generated from repository-local signals

Project-specific bridge generation now uses repository-local content such as docstrings/comments to propose local NL-to-code vocabulary. This keeps the adaptation mechanism available without baking more CodeLens-only aliases into the generic path.

### 3. External benchmark smoke coverage is now part of the public story

The benchmark surface now includes small non-CodeLens repo measurements:

- **Flask** (`20` queries): hybrid `MRR 0.563`, semantic `0.577`
- **curl** (`18` queries): hybrid `MRR 0.623`, semantic `0.555`

These are deliberately framed as **smoke evidence**, not a promotion-grade multi-language matrix. That distinction is part of the release, not a footnote.

### 4. Product documentation matches the actual release surface

The public docs now reflect the current shipping shape:

- workspace version `1.9.14`
- `101` registered tool definitions
- `65 / 101` source-declared output schemas
- published channels: crates.io, GitHub Releases, Homebrew tap, installer script, source build
- published release binaries: `darwin-arm64`, `linux-x86_64`, `windows-x86_64`

### 5. Enterprise productization gates are documented

[ADR-0002](../adr/ADR-0002-enterprise-productization-evaluation-and-release-gates.md) now anchors the product-quality direction:

- separate generic adaptation from project-local adaptation
- require tiered evaluation beyond self datasets
- tighten release packaging and provenance expectations
- keep fast path and precise path explicit

## Current benchmark snapshot

### Internal regression baselines

| Dataset | Semantic MRR | Lexical MRR | Hybrid MRR |
| ------- | -----------: | ----------: | ---------: |
| Self regression (`104` queries) | 0.798 | 0.614 | **0.841** |
| Role regression (`70` queries) | 0.900 | 0.832 | **0.962** |

### External smoke

| Repo | Queries | Semantic MRR | Lexical MRR | Hybrid MRR |
| ---- | ------: | -----------: | ----------: | ---------: |
| Flask | 20 | **0.577** | 0.363 | 0.563 |
| curl | 18 | 0.555 | 0.512 | **0.623** |

Interpretation:

- hybrid is still the promoted default path
- Flask remains a real calibration gap
- external evidence is now published, but it is not yet broad enough for a strong enterprise generalization claim

## Distribution and operations

Current install and deployment channels:

- `cargo install codelens-mcp`
- `brew install mupozg823/tap/codelens-mcp`
- GitHub Releases artifacts
- `install.sh`
- source build via `cargo build --release`

Recommended deployment split:

- `7837` read-only HTTP daemon for planner/reviewer/CI surfaces
- `7838` mutation-enabled HTTP daemon for explicit refactor flows only

The release workflow builds and publishes:

- `darwin-arm64`
- `linux-x86_64`
- `windows-x86_64`

and then updates the Homebrew tap from the published release checksums.

Release verification guide:

- [Release verification and packaging status](../release-verification.md)

cargo check

cargo test -p codelens-mcp --features http
bash scripts/quality-gate.sh --mode local

```bash
python3 benchmarks/token-efficiency.py .
python3 benchmarks/embedding-quality.py . --isolated-copy
```

## Known limits

- External benchmark evidence is still smoke-level, not a promotion-grade Python/JS/JVM/systems matrix.
- The product now distinguishes generic adaptation from project-local bridges, but enterprise claims should still be based on the generic path unless explicitly stated otherwise.
- Hybrid is not universally best on every repo yet; Flask demonstrates that directly.
