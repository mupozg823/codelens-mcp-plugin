# v1.9.31 Release Notes

Dispatch decomposition + agent ergonomics. First post-`v1.9.30` tag that ships behavior changes (previous 1.9.27–1.9.30 were release-plumbing only). API break: none.

## Scope

| Change                                          | Surface                                             | Size                                      |
| ----------------------------------------------- | --------------------------------------------------- | ----------------------------------------- |
| `dispatch/mod.rs` 6-way decomposition           | `crates/codelens-mcp/src/dispatch/`                 | 1,133 → 187 at entry; 5 new sibling files |
| Structured error recovery (`RecoveryHint`)      | `error.rs` + `protocol.rs` + `dispatch/response.rs` | +70 LOC, 7 variants mapped                |
| `reasoning_scaffold` on workflow responses      | `tools/reasoning_scaffold.rs` + `protocol.rs`       | +103 LOC, 6 workflow tools covered        |
| `infer_harness_phase` fallback                  | `tools/mod.rs` + `dispatch/mod.rs`                  | +70 LOC, 7 unit tests                     |
| `WATCHER_RECENT_FAILURE_WINDOW_SECS` relocation | `state.rs` → `state/watcher_health.rs`              | −1 LOC from state root                    |

## Why

1. **God-module debt** — the 4/16 architecture audit left `dispatch/mod.rs` (1,133 lines, mixed responsibilities) as the single biggest remaining structural risk. Splitting it into six single-responsibility files eliminates the hot-spot without touching the external `pub(crate) fn dispatch_tool` contract.
2. **Agent ergonomics** — previously, error messages and workflow responses were strings; downstream agents had to regex or keyword-match to recover. `RecoveryHint` and `reasoning_scaffold` expose the same intent as tagged JSON so harnesses can branch structurally.
3. **Phase-aware composite guidance** — `suggested_next_tools` already weighted `_harness_phase` when set, but most clients never sent the flag. Inferring it from the recent-tool trail (priority `build > review > plan`) makes the existing logic productive for clients that don't set it explicitly.

## Recovery hint mapping

| Error variant           | `RecoveryHint`                                                                     |
| ----------------------- | ---------------------------------------------------------------------------------- |
| `MissingParam(field)`   | `RequireField { field }`                                                           |
| `ToolNotFound(_)`       | `FallbackTool { "get_capabilities", "list available tools" }`                      |
| `FeatureUnavailable(_)` | `RequireFeature { "semantic", "rebuild with --features semantic" }`                |
| `LspNotAttached(_)`     | `FallbackTool { "find_symbol", "tree-sitter satisfies most lookups without LSP" }` |
| `IndexNotReady(_)`      | `RetryAfterSeconds { 5 }`                                                          |
| `Timeout { .. }`        | `FallbackTool { "start_analysis_job", "move heavy work to the durable queue" }`    |
| `ResourceExhausted(_)`  | `RetryAfterSeconds { 10 }`                                                         |

## Reasoning scaffold coverage

Six planner/reviewer workflow tools now emit a structured scaffold alongside the payload. Format: `{ what_this_tells_you, what_this_does_not_tell_you, if_looking_for_X: "run Y next" }`. Tools without a registered scaffold return `None` and the field is omitted from the response.

## Compatibility

- **Public API**: `pub(crate) fn dispatch_tool` is byte-identical in signature. External callers (`server/router.rs`, `server/oneshot.rs`) require no change.
- **Wire format**: `ToolCallResponse` gains two optional fields (`recovery_hint`, `reasoning_scaffold`) with `skip_serializing_if = "Option::is_none"`. Existing clients see no format shift when these fields are absent.
- **Features**: `semantic`, `http`, `scip-backend` all compile green; feature gate paths unchanged.
