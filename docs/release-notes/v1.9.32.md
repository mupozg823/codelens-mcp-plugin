# v1.9.32 Release Notes

Internal refactor only — two more god-module splits. No behavior change, no API break, no test additions. Builds directly on v1.9.31.

## Scope

| Change                   | Before    | After                                      |
| ------------------------ | --------- | ------------------------------------------ |
| `tools/mod.rs` split     | 875 lines | **214** + new `tools/suggestions.rs` (673) |
| `main.rs` CLI extraction | 504 lines | **265** + new `cli.rs` (249)               |

## Why

v1.9.31 cleared the `dispatch/mod.rs` god module (1,133 → 187). The next two outliers in the 2026-04-17 `find_misplaced_code` report were `tools/mod.rs` (composition + recommendation engine mixed) and `main.rs` (bootstrap + CLI parser mixed). Splitting both preserves external API while reducing the "what does this file own" surface area at each entry point.

## `tools/mod.rs` → core + `tools/suggestions.rs`

`tools/suggestions.rs` now owns:

- `PLAN_PHASE_TOOLS` / `BUILD_PHASE_TOOLS` / `REVIEW_PHASE_TOOLS` / `EVAL_PHASE_TOOLS` constants
- `MUTATION_TOOLS` / `REVIEW_TOOLS` / `EXPLORATION_TOOLS` internal filter lists
- `infer_harness_phase` + 7 `phase_inference_tests`
- `suggest_next_contextual`, `composite_guidance_for_chain`, `suggest_next`, `suggestion_reasons_for`
- `is_workflow_tool_name`, `has_recent_low_level_chain`, `composite_suggestions_for_surface` helpers

`tools/mod.rs` keeps only:

- submodule declarations (`pub mod composite`, etc.)
- `tool_registry!` macro + `dispatch_table()`
- `estimate_tokens`, `parse_lsp_args`, `default_lsp_command_for_path`, `default_lsp_args_for_command`

All public entry points (`crate::tools::suggest_next`, `suggest_next_contextual`, `composite_guidance_for_chain`, `suggestion_reasons_for`, `infer_harness_phase`) remain accessible via `pub(crate) use suggestions::*` re-exports. Integration tests that call `crate::tools::suggest_next` compile unchanged.

## `main.rs` → bootstrap-only + `cli.rs`

`cli.rs` now owns:

- `StartupProjectSource` enum + `is_explicit()` / `label()` impls
- `flag_takes_value` (internal)
- `parse_cli_project_arg`
- `select_startup_project_source`
- `resolve_startup_project`
- `cli_option_value`
- `format_http_startup_banner` (HTTP feature-gated)
- `startup_tests` module (6 tests: CLI flag skipping, double-dash separator, `=` form, banner identity fields, fail-closed semantics)

`main.rs` keeps only:

- module declarations
- `use` imports + `pub(crate) use state::AppState`
- `init_tracing_fmt_only` / `init_tracing` (otel feature-gated)
- `fn main` entry point
- `#[cfg(test)] mod tests { #[path = "integration_tests/mod.rs"] }` hook

## Verification

```bash
cargo check --workspace --all-features            # 0 errors, 0 warnings
cargo check -p codelens-mcp --no-default-features # clean
cargo test  -p codelens-mcp                       # 248 passed (unchanged from v1.9.31)
cargo test  -p codelens-engine                    # unchanged, all passed
cargo clippy --workspace --all-features           # 0 new warnings
```

The `analysis_lists_expose_resource_handles_and_counts` test remains timing-flaky under heavy parallel load (polling loop around `get_analysis_job`); passes cleanly on retry or when run in isolation. Flakiness is pre-existing and unrelated to this refactor.

## Compatibility

- **Public API**: no external-facing signature changed in either split.
- **Wire format**: no new fields added; `v1.9.31` scaffold/recovery/phase-inference fields still flow through unchanged.
- **Feature gates**: `semantic`, `http`, `scip-backend`, `otel` all compile green on the split layout.
