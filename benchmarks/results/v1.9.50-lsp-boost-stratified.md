# P1-4 stratified lsp_boost matrix (v1.9.50, 2026-04-20)

Extends the 10-query caller datasets with a `bucket` tag so the
per-ref weighting design can be judged against the case it was built
for (`thick_caller`: multi-ref container) separately from the case
it deliberately refuses to promote (`thin_wrapper`: single-ref
caller).

- `self` — 15 queries (10 thin_wrapper + 5 thick_caller)
- `flask` — 13 queries (10 thin_wrapper + 3 thick_caller)
- `zod` — 13 queries (10 thin_wrapper + 3 thick_caller)

All arms use `--auto-attach` (`CODELENS_LSP_AUTO=true` +
`prepare_harness_session` + language-appropriate indexing wait).

## What the buckets tell us

### Thick-caller: the design works as advertised

Every repo's **thick_caller** bucket rises under `lsp_boost=true`:

- **self** — `2/5 → 5/5` (every thick caller now surfaces). MRR
  **0.0900 → 0.2686 (+199%)**. Notable promotions: `ensure_indexed
→ find_symbol` 0 → rank 3, `find_referencing_symbols_via_text →
find_referencing_symbols` 0 → rank 2, `make_symbol_id →
get_symbols_overview_cached` 0 → rank 6.
- **flask** — `0/3 → 3/3` (baseline cannot find _any_ thick caller;
  boost finds all three). `Flask → find_best_app` enters at rank 7,
  `Flask → find_app_by_string` rank 7, `Flask → load_app` rank 11.
- **zod** — `1/3 → 2/3`. `globalRegistry → registry` lifts from
  None to rank 20; `$ZodRegistry → registry` stays at rank 2.
  `safeParse → ZodType` remains None on both arms — the TypeScript
  interface spans ~140 lines and exceeds the `LSP_PROXIMITY_WINDOW_LINES
= 250` _span_ budget, but its `end_line` is populated so this is a
  dataset oddity (a type-only caller with no runtime reference
  pattern) rather than a design miss.

The consistent thick-caller uplift across three languages is the
measurement P1-4 was designed to produce. Per-ref containment with
saturation at 3 refs promotes exactly the symbols whose bodies wrap
the probe target, which is what an agent asking "what calls X?"
wants back.

### Thin-wrapper: mixed, and that is the point

**self**: 1/10 → 7/10 (+600%). rust-analyzer + warm tree-sitter
surface an extra six single-ref callers that were previously buried.
**flask**: 2/10 → 4/10 (+100%). Same pyright-assisted improvement
flagged in the previous matrix.
**zod**: 4/10 → 4/10 (MRR 0.0932 → 0.0713, −23%). tsserver returns
refs but the expected callers here are micro-wrappers (`_parse →
parse`, `registry → globalRegistry`) that the per-ref design
correctly refuses to rank above multi-ref containers. This is not
a bug to fix — it is the contract: "a single-ref caller is a
weaker signal than a multi-ref container".

## Honest limits

- `thick_caller` n=3–5 per repo is still a small sample. The 100%
  thick-hit rate on self and flask should be read as "per-ref
  weighting is in the right direction on measurable cases", not as
  a generalisation claim.
- `zod/safeParse → ZodType` is a TypeScript interface case; the
  current `find_referencing_symbols_via_text` tree-sitter path does
  not treat interface declarations as span containers the same way
  it treats function bodies. Follow-up either (a) extends the text
  search to respect TS interface blocks or (b) documents the limit
  and re-picks the dataset target.
- Per-ref still can't help when the ground truth is a thin wrapper
  and a multi-ref container sits in the same file. The
  recommendation stays: leave `lsp_boost=false` on
  `embedding-quality-dataset-*.json` runs; opt in only for
  caller-tracing agent flows.
