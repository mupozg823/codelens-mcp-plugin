# P1-4 follow-up: TS interface container rescue (v1.9.50, 2026-04-20)

Extends the stratified matrix (§1c, commit `7d4b292`) with a
container-text-rescue branch that activates when:

1. `text_score == 0` (the candidate name does not overlap the query),
2. `ctx.weights.lsp_signal > 0.0` (P1-4 boost armed),
3. `lsp_proximity >= 0.6` (≈ two refs contained by the candidate span
   under `LSP_CONTAINMENT_SATURATION_REFS = 3`).

When these hold, the text component is synthesised as
`70 * proximity * text_weight` instead of `text_score * text_weight`.
The `70` (not `100`) magnitude is deliberate: the rescue lets the
container compete with partial-name helpers (text_score ≈ 30–50) but
stays below the ceiling of exact-name matches (text_score ≈ 100), so
`safeParse → safeParse` still beats `safeParse → ZodType` when both
live in the same file.

The probe also replaces `find_referencing_symbols_via_text` with
`find_all_word_matches` + a two-pass (prod-first, tests-second)
per-file cap, so large monorepos do not saturate the ref collection
on tests before the container file is reached. This was the
mechanical reason `classic/schemas.ts` never entered the boost set
for zod prior to this commit.

## What the rescue recovers

The zod `safeParse → ZodType` case (previously `rank=None` on both
arms in the §1c stratified matrix) now lifts to **rank 6**. The
interface `ZodType` declares `safeParse` seven times across its body
(lines 58, 60, 196, 255, 256 survive after comment/string filtering);
five of those refs enter the boost set under the two-pass per-file
cap, two are contained within the interface's span
(`20..=151`). Containment factor `2/3 ≈ 0.667 >= 0.6` triggers the
rescue, `text_component = 70 × 0.667 × 0.70 ≈ 32.7`, which is enough
for ZodType to displace type-alias helpers like `SafeParseReturnType`
that share a prefix with the query but contain no body-level refs.

All three repos now hit **every thick_caller row**:
`self 5/5`, `flask 3/3`, `zod 3/3`.

## Where it regresses

`zod` thin_wrapper drops from 4/10 MRR 0.0932 to 3/10 MRR 0.0333.
The rescue does not target thin-wrapper queries directly — their
`expected_symbol` is typically exact-name-match (text_score > 0) — but
unrelated symbols in the same dense file (`core/parse.ts` contains
hundreds of `_parse` occurrences) start clearing the `>= 0.6`
proximity bar and rank above the intended single-ref wrapper.

This is the explicit trade-off of the rescue: recovering ZodType
costs a handful of ranks on dense-mention thin wrappers in the same
repository. The contract from the §1c note holds — per-ref weighting
still refuses to promote thin wrappers over multi-ref containers —
but the rescue tilts the noise slightly more toward multi-ref
containers even when they are _not_ the expected symbol.

Self and flask do not show the thin-wrapper drop because their
single-ref callers are much more scattered (Rust source / Flask
Python modules spread the ref density across files), so the rescue
does not consistently over-lift a competitor in the expected file.

## Honest limits

- The rescue is threshold-sensitive. `>= 0.6` is a compromise; the
  initial implementation at `> 0.3` (single contained ref) regressed
  both flask and zod aggregate MRR, and `>= 1.0` (full saturation, ≥ 3
  refs) leaves `safeParse → ZodType` below the threshold. Future work
  could replace the hard threshold with a smooth ramp.
- `70 × proximity` still caps below an exact-name match. Queries
  where the caller's name shares a prefix with an unrelated helper
  (like `SafeParseReturnType`) can still outrank the true container
  when the helper's text_score exceeds ~47.
- The two-pass per-file cap prioritises non-test files, which helps
  zod-style monorepos but silently ignores test code. For repos
  whose callers legitimately live under `/tests/`, the rescue will
  not fire. This matches the per-ref design intent (caller tracing
  from production code), not a bug.
- Aggregate MRR is not a single-number summary for this feature.
  Bucket-wise hit rates (`thick_caller = 5/5, 3/3, 3/3`) are the
  measurement P1-4 was designed to move; headline MRR is mixed
  because the same repository shows uplift on one bucket and
  slippage on another under the same arm.
