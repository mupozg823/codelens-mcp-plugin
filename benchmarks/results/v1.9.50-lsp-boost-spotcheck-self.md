# P1-4 lsp_boost spot-check (self, v1.9.50, 2026-04-20)

**Binary:** `target/release/codelens-mcp` (built from commit 21bbaac)
**Repo:** `codelens-mcp-plugin` (self)
**LSP:** rust-analyzer (Homebrew)
**Dataset:** 5 hand-crafted queries anchored at known definition files
**Script:** `benchmarks/lsp-boost-spotcheck.py`

## Headline

| Arm       | `lsp_boost` | MRR    | hits | mean elapsed_ms (non-cold) |
| --------- | ----------- | ------ | ---- | -------------------------- |
| baseline  | false       | 1.0000 | 5/5  | ~80                        |
| lsp_boost | true        | 1.0000 | 5/5  | ~120                       |

## Why MRR is identical

The micro-dataset queries each symbol by its own **exact name**
(`rank_symbols`, `prune_to_budget`, ...). The exact match already
scores 93–98 and lands at rank 1 in both arms, so MRR is pinned at 1.0
regardless of what the LSP boost does to secondary candidates. This is
a dataset limitation, not evidence that the probe is inert.

## Evidence the probe actually fires

Query 3 (`get_ranked_context_cached`, anchored at `reader.rs`):

- baseline candidate pool size: **5**
- lsp_boost=true candidate pool size: **6**

The extra entry is `get_ranked_context_cached_with_lsp_boost`, the
newly-added caller in `reader.rs`. The baseline never surfaced it
because the P1-4 boost set was empty. With `lsp_boost=true` the
rust-analyzer probe returned it and the engine pulled it into the
ranked top-5. Secondary ordering shifted; top-1 did not.

Latency also shifts by roughly 50–70 ms per call once caches are warm,
consistent with a real (not silently-failing) LSP round-trip.

## Honest limits

- Dataset is 5 queries on one repo — not a power measurement, just a
  wiring proof.
- Cold-start was absorbed by query #1 (baseline rank_symbols: 1713 ms);
  subsequent queries are warm-cache.
- CLI one-shot spawns a fresh rust-analyzer per query by default; the
  reuse we see is because the LSP session pool lives for the life of
  the single process, not across processes. A persistent HTTP daemon
  would be the proper home for a real matrix.
- Re-ranking top-1 uplift requires queries whose expected answer is a
  **caller** of the anchor symbol, not the anchor itself. Building
  such a dataset is the next follow-up.
