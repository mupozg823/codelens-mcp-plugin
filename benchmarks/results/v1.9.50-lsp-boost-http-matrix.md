# P1-4 HTTP daemon lsp_boost matrix (v1.9.50, 2026-04-20)

HTTP daemon bench harness (`benchmarks/lsp-boost-http-matrix.py`)
that keeps one daemon alive across the whole run and, with
`--warm-lsp`, fires an LSP-only `find_referencing_symbols(use_lsp=true,
union=true)` probe per `(path, query)` before the measurement arms
execute. The warm probe gives rust-analyzer / pyright /
typescript-language-server time to index the workspace before the
real ranked-context call asks it for refs.

All arms use the per-ref ranking release (commit a9e23f1).

## Headline — 3-repo matrix with warm LSP

## Readings

### Flask: warm LSP is the difference

Pyright returns _precise_ caller refs where tree-sitter would also
flag any lexical match. On identifier-dense helpers like `url_for`
(tree-sitter 62 refs vs LSP 1), tree-sitter saturates the
`LSP_CONTAINMENT_SATURATION_REFS = 3` threshold in every function
that even mentions the token, so the per-ref weighting degenerates
into a wash. Warm LSP gives a much tighter ref set, and the expected
caller pulls ahead. `Scaffold → App` stayed a hit; `send_file →
send_from_directory` rose from rank 7 to rank 3; a third query
(`render_template → stream_template`) crossed back into top-K.

### Self: already saturated by tree-sitter

rust-analyzer returned 4–67 refs per query, but the tree-sitter
floor on a fully-indexed Rust tree already matches. The +0.003 MRR
gain from warm LSP is noise.

### Zod: dataset is the limit

tsserver returned real refs too (lsp=1..32 per query) but the
expected-caller design in `lsp-boost-dataset-zod.json` asks the
ranker to prefer a thin one-ref wrapper over legitimate multi-ref
containers. Per-ref weighting is correct by design to prefer the
multi-ref container, and a tighter ref set from LSP does not change
that verdict. The dataset needs "thick caller" expectations before
zod can move.

## What the HTTP harness changed in the measurement stack

1. **Persistent daemon**: single `codelens-mcp --transport http` run
   for the whole matrix, so the LSP session pool is shared across
   queries. CLI one-shot was cold-starting rust-analyzer per call
   and returning 0 refs in ~7 ms.
2. **`--warm-lsp` prewarm**: dedicated `find_referencing_symbols`
   probes with `use_lsp=true, union=true` happen before the arms so
   subsequent `get_ranked_context(lsp_boost=true)` calls hit a warm
   pool.
3. **`structuredContent` extraction**: the adaptive compression
   pipeline may shrink `content[0].text` to a skeleton
   (`{success, truncated, compression_stage, ...}`). The harness now
   reads `result.structuredContent.data.symbols` first and only
   falls back to text when structured output is missing.

## Honest limits

- Datasets are 10 queries each. Direction-only.
- Warm-LSP cost: ~3–5 s per unique `(path, query)` prewarm call.
  The matrix as a whole takes 1–2 minutes per repo. Real-world use
  would piggy-back the prewarm on `prepare_harness_session`'s
  auto-attach rather than running it per call.
- Per-ref weighting still can't paper over a dataset where the
  ground truth is "thin wrapper"; the zod regression is intrinsic.
