**Binary:** `target/release/codelens-mcp` after gate/pool fixes
**Repo:** `codelens-mcp-plugin` (self)
**Refs backend:** LSP (rust-analyzer) + tree-sitter union, per `lsp_boost_probe`
**Dataset:** `benchmarks/lsp-boost-dataset-self.json` — 10 queries where

**Script:** `benchmarks/lsp-boost-spotcheck.py --dataset ...`

## What changed between the no-op spot-check and this matrix

The earlier spot-check (`v1.9.50-lsp-boost-spotcheck-self`) showed
MRR=1.0 in both arms because every query was an exact symbol name —
the definition always pinned rank 1. That hid two real gaps:

1. **Gate dropped rescued callers.** `rank_symbols` filtered out any
   candidate with zero text score before the LSP component was even
   added. Callers never share tokens with the query's target, so
   every rescue was silently pruned.
2. **Candidate pool was scoped to `path` only.** Even after rescuing
   the gate, the pool only contained `path`'s symbols —
   `get_symbols_overview_cached(path)` never looked at the caller
   files the LSP probe returned.

This matrix forced those gaps open. Fixes:

- `rank_symbols` now lets symbols whose file is in `ctx.lsp_boost_files`
  survive the text-match gate _when_ `ctx.weights.lsp_signal > 0.0`.
  Default weight 0.0 keeps the old gate closed.
- `get_ranked_context_cached_with_lsp_boost` now pulls
  `get_symbols_overview_cached(boost_file, depth)` for every path in
  `lsp_boost_files` that is not already `path`, deduplicated by
  symbol ID.
- `lsp_boost_probe` unions LSP with tree-sitter (same policy as
  `find_referencing_symbols` with `union=true`). On CLI one-shot mode
  rust-analyzer is cold and returns 0 refs in ~7 ms; tree-sitter is
  the working floor and is what actually populates the boost set on
  this run.

Two new engine tests pin the gate behaviour:
`lsp_signal_rescues_candidate_with_zero_text_score` +
`lsp_signal_gate_stays_closed_when_weight_is_zero`.

## Honest limits

- **File-level boost is coarse.** Every symbol in a boosted file
  receives the same `+lsp_signal * 100` contribution, so irrelevant
  helpers crowd the expected caller. That is why weight 1.0 pushed
  `rank_symbols`'s caller _down_ (21) versus weight 0.25 (17) on one
  query: the lift was uniform, not targeted. A future per-symbol
  variant (boosting only symbols whose line hit an LSP ref row) would
  resolve this.
- **LSP was inert in CLI mode.** `find_referencing_symbols(use_lsp=true)`
  returned 0 refs in ~7 ms on rust-analyzer; the boost set in this
  matrix is almost entirely tree-sitter-derived. Real LSP uplift
  requires HTTP daemon mode with prewarm — still deferred.
- **Dataset is 10 queries on one repo.** Not a power test, but the
  direction (every non-zero arm > baseline by ~0.015-0.023 MRR) is
  consistent across three weight settings.
- **Expected-symbol convention.** Queries are definition names, anchor
  paths are definition files, expected callers live elsewhere. That
  deliberately probes the caller-wiring effect; exact-name queries
  still land at rank 1 regardless of boost (see earlier spot-check).
